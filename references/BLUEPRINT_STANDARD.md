# Blueprint Standard（技术图纸精简百科）

> 这份文档不是合规清单，而是一份**开发者盲点提醒**。很多项目烂尾，不是因为代码写不好，而是因为开发者**不知道自己不知道**——没听过某些图的名字，所以也无法意识到自己缺了什么。
>
> CodeRail 的 `blueprint_check.py` 会扫描你的项目信号，把你这类项目常受益的几张图推到面前。每种图下面有：解决什么痛、缺了会怎样、长什么样（最小示例）、什么时候不需要它。
>
> vibe coding 像徒手盖房子。小屋子不需要图纸；但复杂工程没有图纸，到中后期结构债会积累放大，到最后没法修也没法查——就像没有 BIM 的工程，管线维护无法执行。

## 怎么用这份文档

- 先跑 `python scripts/blueprint_check.py --target .`，看它推荐哪几张图。
- 不用全画。只画它推荐的、且你觉得戳到痛点的。
- 每张图用 mermaid 或 PlantUML 写进 `docs/`，能进 Git 版本控制。
- 这份文档不评判你画没画、画得好不好——它只让你**知道有这些图、它们解决什么**。

# 1. 用户与交互层（表现层）

聚焦用户体验与前端业务逻辑。由产品经理与前端工程师主导。**只有当项目有 UI（web/移动端）时才相关。**

### User Journey Map（用户路径图）

**解决什么痛**：用户为了达成一个目标（如购买商品），在系统里要跨好几个页面操作，但没人把这整条路径画出来，于是某个环节卡住时不知道影响谁。
**症状**：用户反馈"流程不顺"，但团队说不清是哪一步的问题；转化率低却找不到漏斗断点。
**最小示例**（mermaid）：
```
journey
    title 购买商品
    浏览首页 -> 5: 用户
    搜索商品 -> 4: 用户
    加入购物车 -> 4: 用户
    结算 -> 2: 用户
```
**何时不需要**：无 UI 的项目（CLI 工具、模型推理、API 服务）、单页工具。

### User Flow（用户流程图）

**解决什么痛**：比路径图更精细，包含所有判断分支（登录成功去 A 页面，失败去 B 页面）和异常提示流程。路径图只画"理想路径"，流程图画"所有可能"。
**症状**：异常分支（登录失败、支付超时、权限不足）没人设计，由前端临时补，导致每处异常处理都不一样。
**最小示例**（mermaid）：
```
graph TD
    A[登录页] -->|成功| B[首页]
    A -->|失败| C[错误提示]
    C --> A
    B -->|无权限| D[403页]
```
**何时不需要**：无 UI、无分支逻辑的纯展示页、无鉴权的工具。

### Page Flow / Wireframe Flow（页面流转图）

**解决什么痛**：原型图（Wireframes）画了一堆，但"点这个按钮跳哪个页面"全靠口头说。新人不知道页面之间的连线。
**症状**：UI 改版时漏改某个跳转；产品评审时大家在脑子里猜"这个入口通向哪"。
**最小示例**（mermaid）：
```
graph LR
    首页 -->|点击商品| 详情页
    详情页 -->|立即购买| 结算页
    详情页 -->|加入购物车| 购物车页
```
**何时不需要**：单页应用、无多页面的工具、纯 API。

# 2. 系统与应用架构层（业务逻辑层）

聚焦全局代码结构、服务边界、数据流向。由全栈/架构师主导。**几乎任何正经项目都受益于这一层的至少一张图。**

### System Architecture Diagram（系统架构图）

**解决什么痛**：项目有多个技术层（前端/网关/服务/存储/运维），却没人能一眼说清"哪个组件在哪层、依赖谁"。新人 onboard 靠口口相传。
**症状**：改一个配置，不知道影响哪些层；讨论"加个缓存"时每个人在脑子里画不同的图；出了 bug 要花很久才能定位是哪一层的问题。
**最小示例**（mermaid）：
```
graph TB
    UI[前端] --> API[API 网关]
    API --> SVC[业务服务]
    SVC --> DB[(数据库)]
    SVC --> CACHE[(缓存)]
```
**何时不需要**：单文件脚本、无分层的纯 CLI 工具。**几乎所有多组件项目都需要。**

### Component Diagram（模块/功能拓扑图）

**解决什么痛**：项目代码内部有多个模块（前端组件树、后端微服务、Python 子包），但模块之间的调用关系没人画。改一个模块不知道牵连谁。
**症状**：改 A 模块，B 模块莫名挂掉，测试才发现；重构时不敢动，因为不知道依赖图。
**最小示例**（mermaid）：
```
graph LR
    gpt[gpt 模块] --> vqvae[vqvae]
    s2mel[s2mel] --> BigVGAN[BigVGAN]
    infer[infer] --> gpt
    infer --> vqvae
```
**何时不需要**：单模块项目、模块数 < 3。模块越多这张图越值钱。

### Sequence Diagram（时序图/序列图）

**解决什么痛**：前后端接口交互、或服务间调用，按时间顺序的完整链路没人画清。"前端 → 网关 → 权限 → Service → DB → 返回"全靠猜。
**症状**：接口文档写了字段却没写交互顺序；异步流程（先返回后处理）的时序混乱；调试跨服务 bug 时不知道哪一步先发生。
**最小示例**（mermaid）：
```
sequenceDiagram
    前端->>网关: POST /api/order
    网关->>权限: 校验
    权限-->>网关: ok
    网关->>Service: 创建订单
    Service->>DB: INSERT
    DB-->>Service: ok
    Service-->>前端: 200
```
**何时不需要**：无跨组件调用、纯本地计算、单进程无网络交互。

### State Machine Diagram（接口状态机图）

**解决什么痛**：业务对象有复杂状态转化（订单：待付款→待发货→运输中→已签收），但触发条件和异常回滚没人定义清。
**症状**：状态字段加了，但"从 A 能不能到 B"靠代码 if-else 隐式实现，没人能说全；出现非法状态转换（如已签收又变待发货）才发现漏了校验。
**最小示例**（mermaid）：
```
stateDiagram-v2
    [*] --> 待付款
    待付款 --> 待发货: 支付成功
    待发货 --> 运输中: 发货
    运输中 --> 已签收: 确认
    待付款 --> 已取消: 超时
```
**何时不需要**：无状态转化的业务（纯计算、纯推理、无生命周期的 CRUD）。

# 3. 数据与模型层（持久化层）

聚焦结构化数据的存储与流动。由后端与 DBA 主导。**只有当项目有数据库或复杂数据流时才相关。**

### ER Diagram（实体关系图）

**解决什么痛**：数据库有好多表，表与表的关联（1对1、1对多、多对多）没人画清。字段、主外键关系全埋在 migration 文件里。
**症状**：改一张表不知道哪些表的外键指向它；写 join 查询时要翻好几个文件才搞清关系；新人不知道"用户和订单是怎么关联的"。
**最小示例**（mermaid）：
```
erDiagram
    USER ||--o{ ORDER : places
    ORDER ||--|{ LINE_ITEM : contains
    USER {
        int id PK
        string name
    }
```
**何时不需要**：无数据库的项目（纯内存计算、文件存储、模型推理）。

### Data Flow Diagram（数据流图 DFD）

**解决什么痛**：系统里数据从哪来、经过哪些加工、存到哪、输出到哪，这条链路没人画。脱离具体代码，单纯描述数据的流动。
**症状**：大数据处理或复杂报表系统，数据来源多、加工步骤多，出了问题不知道是哪一步算错；新增数据源时不知道接入哪一层。
**最小示例**（mermaid）：
```
graph LR
    A[日志源] --> B(清洗)
    B --> C[(数据仓库)]
    C --> D(聚合)
    D --> E[报表]
```
**何时不需要**：单数据源、无加工链路、简单的增删改查。

# 4. 部署与运维层（基础设施层）

聚焦服务如何变成线上实体。由 DevOps 主导。**只有当项目要部署上线（Docker/k8s/CI）时才相关。**

### Deployment Diagram（部署架构图）

**解决什么痛**：软件在线上的物理分布没人画清——负载均衡、Web 服务器、数据库集群、防火墙（VPC）的拓扑全靠脑子记。
**症状**：线上故障时不知道请求经过了哪些节点；扩容时不知道加在哪一层；安全审计时画不出网络边界。
**最小示例**（mermaid）：
```
graph TB
    用户 --> LB[负载均衡 Nginx]
    LB --> W1[Web 服务 1]
    LB --> W2[Web 服务 2]
    W1 --> DB[(数据库主)]
    W2 --> DB
    DB --> SR[(从库)]
```
**何时不需要**：纯本地运行、无部署、单机工具。

### CI/CD Pipeline Diagram（流水线图）

**解决什么痛**：代码从提交到上线的自动化过程（commit→扫描→测试→构建镜像→灰度发布）没人画清，每个节点的失败处理靠经验。
**症状**：CI 挂了不知道是哪一步；新人不知道"我提交后会发生什么"；发布事故后复盘不出是流水线哪环漏了。
**最小示例**（mermaid）：
```
graph LR
    Commit --> Scan[代码扫描]
    Scan --> Test[单元测试]
    Test --> Build[Docker 镜像]
    Build --> Stage[灰度]
    Stage --> Prod[生产]
```
**何时不需要**：无 CI/CD、手动部署、个人项目。

# 工具与规范建议

| 图类型 | 推荐工具 | 建议 |
|---|---|---|
| 用户路径/流程/页面流 | Figma、墨刀、Axure | 优先保证业务闭环，不追求技术细节 |
| 时序图/架构图/ER图/状态机 | mermaid（进 Git）、PlantUML、draw.io | **优先 mermaid/PlantUML**——纯文本，能进版本控制，和代码一起 review |
| 部署/网络图 | Lucidchart、各云厂商官方图标 | 直接用云厂商（阿里云/AWS）的官方组件 |

**为什么推荐 mermaid/PlantUML**：它们用纯文本（Markdown 友好）生成图，能完美纳入 Git 版本控制。这意味着图纸变更会进 code review，能和 K7 Trace Graph 联动（trace event 的 `modifies` 字段可以指向 `docs/architecture.mmd`）。

# 与 CodeRail 的关系

Blueprint Awareness **不是** CodeRail 内核的一部分（不占 K0–K7 任何一行）。它是 doctor 的一个**非阻塞提醒 section** + 一份参考百科。

- **不门禁**：缺图不阻塞工作，不报 severe，不改 exit code。
- **不评判**：不查你画没画过，只推送"你可能用得上"。
- **不替你画**：画图是设计工作，CodeRail 只让你知道有这些图。
- **自适应**：按项目信号（有 UI？DB？多模块？部署？）推断哪些图相关，不对无 UI 的项目要求 User Flow，不对无 DB 的项目要求 ER 图。

BIM 类比的对应：CodeRail 不帮你画图纸，但它告诉你"这栋楼按它的类型，通常该有这几张图，你听说没？"——把"不知道的不知道"变成"至少听说过"。
