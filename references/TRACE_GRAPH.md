# Trace Graph Reference (K7)

The Trace Graph Kernel gives every meaningful development action a source, a
target, a modification, a validation, and a persistence link.

One-line rule: **No action without a trace link.**

中文: Trace Graph 让每次关键动作都有来源、目标、修改、验证和沉淀位置。

## Purpose

Without trace, an agent's work decays into a stream of edits that no later agent
can audit. With trace, a new agent can ask "why does this file look like this?"
and follow a chain of events back to a goal, a decision, a verify step, and a
handoff.

The trace graph is intentionally small. It is a JSONL append log plus a
generated index. It is not a graph database.

## Trace event types

Only these types are valid. Do not invent more without code already needing them.

- `intent` — a new request or goal appears.
- `align` — a North-Star alignment was performed.
- `task` — a task was created or updated.
- `decision` — a durable engineering decision was made.
- `research` — an investigation, spike, or PoC started.
- `attempt` — an implementation attempt started.
- `change` — files were modified.
- `verify` — a harness, test, build, or manual acceptance was run.
- `handoff` — work was handed off (H1/H2/H3).
- `lesson` — a reusable failure lesson was recorded.

## Trace edge types

Only these edges are valid. Use them as plural list fields on the event.

- `serves` — this event serves a North Star.
- `derived_from` — derived from another event or source.
- `implements` — implements a task.
- `modifies` — modifies a file or asset.
- `validated_by` — validated by a verify event or evidence ref.
- `depends_on` — depends on another event or task.
- `supersedes` — supersedes an earlier event.
- `blocks` — blocks another event or task.
- `relates_to` — loosely related, not a stronger edge.

## JSONL schema

`docs/TRACELOG.jsonl` is append-only JSON Lines. JSONL cannot carry comments;
all explanation lives in this file and in `docs/TRACE_INDEX.md`. Keep each
event's `summary` to 1-3 lines. Never paste full logs, full diffs, or full
transcripts.

Common fields:

```json
{
  "id": "TR-20260706-143012-9f3a",
  "ts": "2026-07-06T14:30:12+00:00",
  "type": "change",
  "summary": "Update task template to include CodeRail Coordinate",
  "task": "T-001",
  "north_star": "NS-001",
  "status": "open",
  "source_kind": "user",
  "source_ref": "chat 2026-07-06",
  "files": ["docs/TASKS.md"],
  "serves": ["NS-001"],
  "implements": ["T-001"],
  "modifies": ["docs/TASKS.md"],
  "validated_by": [],
  "depends_on": [],
  "derived_from": [],
  "supersedes": [],
  "blocks": [],
  "relates_to": [],
  "harness_command": null,
  "harness_result": null,
  "commit": null,
  "coordinate": {
    "goal": "Initialize CodeRail-governed project",
    "task": "Create project governance docs",
    "scope": { "allowed": ["docs/**"], "forbidden": ["src/**"] },
    "verify": ["manual template check"],
    "stop": ["business implementation requested"],
    "persist": ["TASKS", "TRACE"]
  }
}
```

Field rules:

- `id`, `ts`, `type`, `summary` are required on every event.
- `id` is auto-generated by `trace_event.py` as `TR-YYYYMMDD-HHMMSS-XXXX`.
- `task`, `north_star`, `files`, and edge fields are optional but recommended.
- A `change` event must have at least one of `task`, `north_star`, or `files`.
- A `verify` event must have `harness_result` (for example `passed` / `failed`).
- `coordinate` is optional on legacy events and recommended on new `change`,
  `verify`, and `handoff` events.
- `status` is free-form but typical values are `open`, `accepted`, `rejected`,
  `superseded`, `blocked`, `done`.

## Usage example

```bash
python3 scripts/trace_event.py \
  --target /path/to/repo \
  --type task \
  --summary "Create initial task contract" \
  --task T-001 --north-star NS-001 --serves NS-001

python3 scripts/trace_event.py \
  --target /path/to/repo \
  --type change \
  --summary "Update task template" \
  --task T-001 --north-star NS-001 \
  --files docs/TASKS.md --implements T-001 --modifies docs/TASKS.md

python3 scripts/trace_event.py \
  --target /path/to/repo \
  --type verify \
  --summary "Template check passed" \
  --task T-001 --north-star NS-001 \
  --harness-command "manual template check" --harness-result passed \
  --validated-by T-001

python3 scripts/trace_index.py --target /path/to/repo
python3 scripts/trace_doctor.py --target /path/to/repo
```

## When you must write a trace

- A task is completed (`done`).
- A user adds a new requirement mid-session.
- A task jumps to a different goal or scope.
- Research or a PoC ends (with `accepted`, `rejected`, or `superseded` status).
- A decision is recorded.
- A handoff is written (H1/H2/H3).
- A harness failure is repaired (verify + lesson).
- An asset is promoted (A0/A1/A2 → A3/A4/A5) or rejected.

## What you must not put in a trace

- Full chat transcripts.
- Full terminal logs.
- Full git diffs.
- Long background explanations.
- Research drafts treated as accepted decisions.
- Trivial micro-edits that need no audit trail.

## Design boundary

CodeRail Trace does **not**:

1. build a graph database;
2. replace a project management system;
3. act as a knowledge base;
4. store full chat records;
5. store full terminal logs;
6. store full git diffs;
7. require users to hand-author complex graph relationships;
8. force a trace on every trivial edit.

CodeRail Trace **only**:

1. records key actions;
2. links source, task, file, test, decision, and handoff lightly;
3. surfaces orphans, drift, and verification gaps;
4. helps the next agent quickly understand why the project looks the way it does.
